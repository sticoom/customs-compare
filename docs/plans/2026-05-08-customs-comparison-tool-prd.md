# PRD：报关单 vs 预录单智能比对工具

## 1. 项目背景

### 1.1 业务场景
在出口报关流程中，需要核对「报关单」（海关盖章版）和「预录单」（系统预录入版）的一致性。目前依赖人工逐字段比对，效率低、易出错。

### 1.2 目标用户
- 报关员 / 单证人员：日常核单使用
- 通过链接分享给同事使用的 Web 工具

### 1.3 核心价值
- 自动提取 PDF 中的字段值
- 按规则自动比对并标记差异
- 减少人工核单时间，降低出错率

## 2. 技术方案

### 2.1 技术路线：规则优先 + AI 兜底

```
核心思路：先用免费的文本提取+正则规则解析字段，解析失败的字段才调用AI辅助
```

**优先级分层：**
1. **第一层：PyMuPDF 文本提取 + 正则规则**（免费，0 tokens，覆盖 90%+ 字段）
2. **第二层：AI 文本模型辅助**（仅处理第一层解析失败的字段，消耗极少 tokens）
3. **第三层：AI 视觉模型兜底**（仅当文本提取完全失败时，将相关页面转图像发送 AI）

### 2.3 技术栈
| 层级 | 技术 | 说明 |
|------|------|------|
| 前端+后端 | **Streamlit** | Python 全栈，内置文件上传、表格展示、布局组件 |
| PDF处理 | PyMuPDF (fitz) | 文本提取（主要方式），图像转换（兜底方式） |
| 正则解析 | Python re | 结构化字段提取，覆盖大部分字段 |
| AI辅助(文本) | 可配置（默认GLM-4-Flash免费） | 解析失败字段辅助，支持切换 DeepSeek/OpenAI 等 |
| AI兜底(视觉) | 可配置（默认GLM-4V-Flash免费） | 文本提取完全失败时使用，支持切换 DeepSeek-VL2 等 |
| Excel导出 | openpyxl | 生成比对报告 |
| 部署 | **Streamlit Cloud** | 免费，从 GitHub 自动部署，生成分享链接 |

### 2.4 Token 消耗预估

| 场景 | 方案 | 预估 tokens | 成本 |
|------|------|-----------|------|
| 大部分字段 | 文本提取+正则 | **0** | **免费** |
| 少数失败字段 | GLM-4-Flash 文本辅助 | ~500-2000 | **免费** |
| 极端情况 | GLM-4V-Flash 视觉兜底 | ~5000-10000 | **免费** |

> GLM-4-Flash 和 GLM-4V-Flash 都是智谱提供的免费模型，完全满足本项目需求。

### 2.5 部署方案

2. [share.streamlit.io](https://share.streamlit.io) 连接 GitHub 仓库
3. 自动构建部署，生成 `xxx.streamlit.app` 分享链接
4. 无需服务器，无需 Docker，免费

**备选方案：** 阿里云函数计算 / 轻量应用服务器（如需自定义域名或更大并发）

## 3. 功能需求

### 3.1 输入区

#### 上传面板
- **报关单上传区**：支持拖拽/点击上传 PDF，可上传多个文件（一个 PDF 可能含多页报关资料）
- **预录单上传区**：支持拖拽/点击上传 PDF，可上传多个文件
- **一键开始比对按钮**

#### 输入处理流程
1. 用户上传报关资料 PDF（含报关单+合同+装箱单+发票等多页）
2. 用户上传预录单 PDF
3. 系统用 PyMuPDF 提取每页文本
4. 关键词自动识别文档类型：
   - 含"中华人民共和国海关出口货物报关单" 且 出口口岸字段值为`-`（空） → **报关单**
   - 含"出境关别" 且 出境关别值不为空（如"大鹏海关"） → **预录单**
   - 含"合同" + "CONTRACT" → **合同页**（提取买方字段）
   - 含"装箱单" → **装箱单**（备用）
   - 含"发票" + "INVOICE" → **发票页**（备用）
5. 正则规则提取各字段值
6. 按合同协议号自动匹配报关单和预录单配对
7. 规则比对引擎执行校验
8. 展示结果

| 序号 | 字段名(报关单) | 字段名(预录单) | 校验类型 | 校验逻辑 |
|------|--------------|--------------|---------|---------|
| 1 | 商品编码 | 商品编码 | 匹配 | 精确匹配 |

| 7 | — | 原产国（地区） | 固定 | 预录单必须为：中国（CHN） |

#### 特殊规则说明

**商品名称及规格型号比对规则：**
1. 商品名称在第一行，精确匹配
2. 规格型号在第二行，以 `|` 分隔为多个申报要素
3. 常规有 5-9 个申报要素，分隔符一般 5-8 个
4. 报关单的规格型号中有特殊映射：
   - `境内自主品牌` 对应预录单的 `1`
   - `不确定是否享惠` 对应预录单的 `2`
5. 如遇到 9 个要素时，最后一项做模糊匹配（关键字段即可）
6. 规格型号中的具体数值（如尺寸、装箱数）需精确匹配

**境内货源地映射表：**
预录单中是 `(代码)城市名` 格式，报关单中是城市名。比对时按城市关键字匹配：
```
东莞市→东莞, 金华市→金华, 江门市→江门, 宁波市→宁波, 台州市→台州,
滨州市→滨州, 上海市→上海, 菏泽市→菏泽, 南平市→南平, 佛山市→佛山,
中山市→中山, 温州市→温州, 临沂市→临沂, 深圳特区→深圳特区, 惠州市→惠州,
广州市→广州, 衢州市→衢州, 玉林市→玉林, 泉州市→泉州, 丽水市→丽水,
河源市→河源, 上饶市→上饶
```

**数量及单位行交换规则：**
- 报关单格式：`36件` / `113千克`（第1行=成交单位，第2行=法定单位）
- 预录单格式：`113千克` / `72个` / `36件`（第1行=法定重量，第2行=第二法定，第3行=成交单位）
- 比对关系：报关单第1行 ↔ 预录单第3行（成交单位），报关单第2行 ↔ 预录单第1行（法定单位）

### 4.2 交互细节

1. **上传区**：左右并排两个 `st.file_uploader`，报关单区（左）和预录单区（右），各自支持多文件
2. **解析进度**：`st.progress` + `st.status` 显示当前步骤
3. **结果区**：
   - 顶部概览条：合同协议号 + 统计数字（`st.metric`）
   - 表头字段：`st.dataframe` 左右对照，颜色标记状态
   - 商品明细：每个项号一个 `st.expander`，可展开/收起
   - 人工确认项：独立 `st.expander`，带确认按钮
4. **差异高亮**：用 Streamlit 的 `style` 给 dataframe 单元格上色
5. **导出Excel**：`st.download_button` 一键下载

## 5. 数据流

```
1. 用户上传报关资料PDF + 预录单PDF
          │
2. PyMuPDF 提取每页原始文本（免费，0 tokens）
          │
3. 关键词识别每页文档类型
   ├─ "海关出口货物报关单" + 出口口岸值为"-" → 报关单页
   ├─ "出境关别" 值不为空（如"大鹏海关"） → 预录单页
   ├─ "合同" + "CONTRACT" → 合同页（提取买方）
   └─ 其他 → 装箱单/发票（备用）
          │
4. 正则规则提取字段（免费，0 tokens）
   - 大部分字段直接匹配成功
   - 记录解析失败的字段
          │
5. [如有失败字段] AI 文本辅助（~500 tokens）
   - 只发送失败字段的上下文文本片段
   - GLM-4-Flash 辅助提取
          │
6. [如仍然失败] AI 视觉兜底（~5000 tokens）
   - 只转换相关页面为图像
   - GLM-4V-Flash 识别
          │
7. 按合同协议号匹配报关单和预录单
          │
8. 规则比对引擎逐字段执行校验
   - 精确匹配 / 固定值校验 / 模糊匹配
   - 标记比对结果状态
          │
9. Streamlit 渲染结果 + 可导出 Excel
```
